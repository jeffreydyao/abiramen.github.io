CSE and SSHFS
Stop killing CSE servers with this one simple trick!
Tags: unsw-cse guide ssh
keyboard_arrow_down

If you‚Äôre a VS Code user who is in first year, or isn‚Äôt too used to coding locally, you might be better off checking the easier method of using SSHFS with VS Code here.

Hello! About a year and a half ago, I wrote a blogpost about setting up Visual Studio Code to connect to CSE machines using the ‚ÄòRemote - SSH‚Äô extension.

In hindsight, this was a terrible mistake with consequences I could not have foreseen. It turns out that the extension spawns a nodejs server on the remote machine for each connection, which requires an intense amount of resources 1.

CSE also has a lot of students, meaning you are not going to be having a good day if your hostname is corelli or mandolin üìâüìâ2. I apologise for the terrible atrocities which I have inflicted upon the servers of the School of Computer Science and Engineering üôèü•∫.

Anyhow, this guide steps through setting up SSHFS to mount and access your CSE filesystem from your local file system! Doing so allows you to access files on CSE machines as if they are on your own computer, and automatically synchronises any changes to files. This method likely uses less than a tenth of the resources required by the VS Code remote extension!

Who is this guide for?

If you‚Äôve reached the end of the first year of your CSE degree, you probably should have a local coding setup by now, whether that be:

  • a native Linux (or *nix) setup, or
  • some form of macOS setup, or
  • WSL 2 3

In that case, I highly recommend using SSHFS if you find yourself needing to work with files directly on CSE machines. If you‚Äôre in first year and feeling adventurous, you can take a try at setting it up too, but you‚Äôre probably better off following this guide. 4

Now that that‚Äôs out of the way, we‚Äôll need to install SSHFS to get started.

Linux and WSL 2

I‚Äôm going to assume that you‚Äôre using an Ubuntu or Debian-based installation (this is most WSL distributions by default), in which case this should be nice and simple. If you‚Äôre not, you‚Äôre probably familiar enough with your distro‚Äôs package manager, or can grab the installer from GitHub here.

Otherwise, all you‚Äôll need to do is open a terminal and run sudo apt install -y openssh-client sshfs on your own computer.

macOS

  1. Install Homebrew if you don‚Äôt have it already by following the instructions here.
  2. Run brew install sshfs in your terminal and follow any steps which pop up.

Mounting your CSE home directory

Our goal is to be able to access our CSE files from our own computer. To do this, we‚Äôll need to use SSHFS to mount them.

  1. If this is your first time, make a directory in your local home directory by running mkdir ~/cse from a terminal. If you want to use a different folder name, feel free to do so, but make sure to use it instead in any future commands. It‚Äôs important that this is an empty folder.
  2. Run the following command: sshfs -o idmap=user -C z5555555@loginX.cse.unsw.edu.au: ~/cse, replacing with your own zID, and replacing the X with the last digit of your zID. For example, if your zID was z5123456, you‚Äôd use login6.cse.unsw.edu.au.

Yay! Your CSE files should now be accessible in the ~/cse directory! You can now access them with the text editor of your choice! For example, if you‚Äôre a VS Code user, you can run code ~/cse to open a new workspace 5.

Once you‚Äôre done working, you should unmount the CSE filesystem. To do this, run fusermount -u ~/cse 6.

In order to get a shell on CSE machines and run commands such as autotest and give, you‚Äôll need to run ssh z5555555@loginX.cse.unsw.edu.au, (again, with your own zID, and replacing X with the last digit of your zID) and type in ‚Äòyes‚Äô if any prompts appear. Use Ctrl+D to exit when you‚Äôre done.

Setting up SSH keys

If you‚Äôre being prompted for your password whenever using SSHFS or SSH, setting up a pair of SSH keys (if you haven‚Äôt already) is probably a good idea. You can do this by following Part 4 and 5 of the ‚ÄòSSH and SSH keys‚Äô section here.

Aliases

Typing in all these commands gets a bit cumbersome. We can use aliases to simplify this process a bit for us!

Determining your shell

Before we can figure out where to keep our aliases, we‚Äôll need to figure out what shell we‚Äôre doing. In order to do this, run echo $SHELL.

It‚Äôs likely that your shell is either bash or zsh (if not, you‚Äôre probably cluey enough to figure the rest out yourself). If you‚Äôre on bash, open or create the file ~/.bashrc. If you‚Äôre on zsh, open or create ~/.zshrc in the text editor of your choice (eg. code ~/.zshrc).

Adding our aliases

Create a newline at the end of the file you just opened, and paste in the following, being sure to replace with your own zID (also make sure to not add a space after the equals sign):

# Your zID (change this)
_SSHFS_ZID=z5555555
# Your desired mountpoint for your CSE home directory
_SSHFS_CSE_MOUNT="$HOME/cse"

alias csemnt="sshfs -o idmap=user -C ${_SSHFS_ZID}@login${_SSHFS_ZID: -1}.cse.unsw.edu.au: ${_SSHFS_CSE_MOUNT}"
alias cseumount="fusermount -zu ${_SSHFS_CSE_MOUNT}"

function cse() {
    # determine where we are relative to the mountpoint (thanks @ralismark)
    local rel=${PWD##${_SSHFS_CSE_MOUNT}}

    if [ -z "$1" ]; then
        # if we don't have arguments, we give the user a shell on the remote cse machine.
        if [ "$PWD" = "$rel" ]; then
            # in the case that we're not in our mountpoint, provide a shell in their home directory.
            ssh "${_SSHFS_ZID}@login${_SSHFS_ZID: -1}.cse.unsw.edu.au"
        else
            # if within the mountpoint, cd to the equivalent dir on the remote before providing a shell (thanks @ralismark)
            ssh "${_SSHFS_ZID}@login${_SSHFS_ZID: -1}.cse.unsw.edu.au" -t "cd $(printf "%q" "./$rel"); exec \$SHELL -l"
        fi
    else
        # if we have arguments, we have a command to execute.
        if [ "$PWD" = "$rel" ]; then
            # in the case that we're not in our mountpoint, we'll execute in the home directory.
            ssh -qt "${_SSHFS_ZID}@login${_SSHFS_ZID: -1}.cse.unsw.edu.au" "$@"
        else
            # if within the mountpoint, cd to the equivalent dir on the remote before executing (thanks @ralismark)
            ssh "${_SSHFS_ZID}@login${_SSHFS_ZID: -1}.cse.unsw.edu.au" -qt "cd $(printf "%q" "./$rel") && $(printf "%q " "$@")"
        fi
    fi
}

You should now be able to open a new terminal (or run exec bash or exec zsh depending on which shell you use) and be able to enter csemnt to mount your CSE filesystem, and cseumount to unmount. You can also use cse to get a shell on CSE machines, and even directly run a single command by using cse 1521 spim -f factorial.s for example. It‚Äôs worth noting that Git operations are particularly slow through SSH FS, so it‚Äôs better to get a shell using cse and perform all your Git operations there.

If this doesn‚Äôt work, and you‚Äôre on bash, make sure to check the relevant part of the troubleshooting section below.

Troubleshooting

mountpoint not empty

This suggests that you‚Äôre trying to mount to the ~/cse directory, despite it currently containing files in it, or perhaps being already mounted. If it‚Äôs the former, remove these files or move them somewhere else if you need them.

Code.exe: Invalid argument

There‚Äôs a weird WSL bug where the code command may not work if you‚Äôre in the ~/cse directory, so you may need to run it from outside - that is, try cd ~ && code ~/cse/path/to/file-or-directory.

Aliases in .bashrc don‚Äôt work

There‚Äôs a chance that your terminal isn‚Äôt set up to automatically source .bashrc. To fix this, open ~/.profile, add a newline to the end of the file and paste the following in:

if [ -n "$BASH_VERSION" ]; then
    if [ -f "$HOME/.bashrc" ]; then
    source "$HOME/.bashrc"
    fi
fi
